## Story

**As a** operador que configura um agente AIOS via deployer,
**I want** que a ferramenta 14-gateway-config.sh consolide todos os dados coletados pelas ferramentas anteriores e gere os artefatos finais de configuracao (`aiosbot.json`, `node.json`, `.env`, `mcp-config.json`),
**so that** o agente tenha todos os arquivos de configuracao prontos para uso imediato, equivalentes ao stack de referencia (`apps/aiosbot-vps-stack-master/`), sem necessidade de edicao manual.

## Acceptance Criteria

1. `14-gateway-config.sh` criado seguindo padrao SetupOrion (source libs, `setup_trap()`, `step_init()`, `log_init()`, `set -euo pipefail`)
2. Le dados de TODOS os `dados_*` em `$STATE_DIR` (pelo menos: dados_vps, dados_portainer, dados_tailscale, dados_openclaw, dados_whitelabel, dados_workspace, dados_llm_router, dados_skills, dados_seguranca, dados_evolution, dados_bridge) â€” com graceful skip para dados opcionais ausentes
3. Gera `aiosbot.json` completo com secoes: meta, env.vars, browser, models (provider `anthropic-router` + fallbacks), agents.defaults (model, workspace, memorySearch, compaction, heartbeat, subagents), tools (shell denyPatterns, filesystem), hooks, channels, gateway, skills, plugins â€” estrutura equivalente ao template do stack de referencia (235 linhas)
4. Models section inclui provider `anthropic-router` com `baseUrl: http://localhost:55119/v1` + ate 11 model aliases (reasoning, media, fast, haiku, sonnet, opus, smart, backup, code, free, gemini3) â€” apenas aliases com API keys disponiveis sao incluidos
5. Tools.shell.denyPatterns preenchido a partir de `dados_seguranca` (blocklist) â€” se indisponivel, usa defaults do stack de referencia (15 patterns)
6. Channels.whatsapp preenchido a partir de `dados_evolution` (se disponivel) â€” se indisponivel, secao omitida ou desabilitada
7. Gateway config com port de `dados_openclaw` (default: 18789), auth password (novo input ou gerado via `openssl rand`), tailscale mode de `dados_tailscale`
8. Gera `node.json` com UUID auto-gerado (`uuidgen`), displayName de `dados_vps` (nome_servidor), gateway host de `dados_tailscale` (hostname.tailnet.ts.net)
9. Gera `.env` consolidado com todas as variaveis organizadas por categoria (Identity, VPS, Locale, LLM Keys, Supabase, Services, Advanced) â€” equivalente ao `.env.example` do stack (50+ linhas)
10. Gera `mcp-config.json` com Brave Search (se BRAVE_API_KEY disponivel), Filesystem (workspace path), Memory (se Supabase configurado) â€” estrutura equivalente ao stack de referencia (23 linhas)
11. Todos os arquivos gerados com `chmod 600` (contem credenciais)
12. Estado salvo em `dados_gateway_config` (paths dos 4 arquivos gerados, status, timestamp)
13. `resumo_final()` exibe tabela com artefatos gerados, paths e confirmacao de permissoes

### Estrategia de Geracao JSON (CRITICO)

O `aiosbot.json` tem 235 linhas com JSON complexo (nested objects, arrays, strings multi-linha como o `compaction.memoryFlush.prompt`). **Heredocs bash sao inadequados** para JSON desta complexidade por causa de:
- Escaping de aspas duplas dentro de aspas duplas
- Caracteres especiais (`\n`, `\t`, regex patterns) no prompt de compaction
- Interpolacao de variaveis bash vs strings literais JSON

**Abordagem recomendada:** Usar `node -e` para gerar o JSON programaticamente:

```bash
# Exemplo: gerar aiosbot.json via node
node -e "
const config = {
  meta: { lastTouchedVersion: '2026.1.24-3', lastTouchedAt: new Date().toISOString() },
  env: { vars: {} },
  // ... montar objeto completo com variaveis do bash passadas via env
};
// Adicionar env vars condicionalmente
if (process.env.OPENROUTER_KEY) config.env.vars.OPENROUTER_API_KEY = process.env.OPENROUTER_KEY;
// Escrever com indentacao
require('fs').writeFileSync(process.env.OUTPUT_PATH, JSON.stringify(config, null, 4));
"
```

**Alternativa:** Usar template file (`aiosbot.json.template`) com placeholders e `sed` para substituicao. Mais simples mas menos flexivel para secoes condicionais (model aliases).

**Para `node.json` e `mcp-config.json`** (simples, <25 linhas): heredoc bash e aceitavel.

[Source: Analise de risco PO â€” complexidade de geracao JSON vs heredoc bash]

### Secoes aiosbot.json Omitidas (Deliberado)

As seguintes secoes do template de referencia sao incluidas como config estatica sem mapeamento dinamico:
- `wizard` â€” auto-gerado com versao e timestamp (mesma logica do `meta`)
- `messages` â€” config estatica (ackReactionScope, tts)
- `commands` â€” config estatica (`{ "native": "auto", "nativeSkills": "auto" }`)
- `bindings` â€” array vazio (`[]`)

Estas secoes devem ser incluidas no JSON gerado usando valores default do stack de referencia.

[Source: apps/aiosbot-vps-stack-master/vps/config/aiosbot.json.template#wizard,messages,commands,bindings]

### Complexidade e Riscos

| Risco | Probabilidade | Impacto | Mitigacao |
|-------|--------------|---------|-----------|
| aiosbot.json incompativel com gateway | Baixa | Alto | Comparar output com template do stack |
| JSON invalido por escaping em heredocs | Media | Alto | Validacao `node -e JSON.parse` apos geracao |
| .env com credenciais expostas | Baixa | Alto | chmod 600 + alerta em hint |
| dados_* ausentes quebram script | Media | Medio | Graceful skip para dados opcionais |
| compaction prompt com caracteres especiais no heredoc | Media | Alto | Usar arquivo auxiliar ou base64 encode para prompt longo |
