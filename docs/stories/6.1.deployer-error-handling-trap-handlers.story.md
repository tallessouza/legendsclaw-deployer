## Story

**As a** operador,
**I want** que qualquer falha durante a instalacao seja capturada com cleanup apropriado,
**so that** o sistema nunca fique em estado parcial/inconsistente e eu saiba exatamente o que falhou.

## Acceptance Criteria

1. Trap handler (`trap 'cleanup' EXIT INT TERM`) presente em TODOS os ferramentas (01-14) e setup.sh
2. Cada ferramentas tem funcao `cleanup_on_fail()` que exibe mensagem de erro com referencia ao log, chama `resumo_final`, e oferece rollback quando aplicavel
3. `deployer.sh` verifica exit code de cada ferramentas executada e mostra feedback adequado (FAIL com log path, nao silencia)
4. `set -euo pipefail` presente no topo de TODOS os scripts (ferramentas + libs executaveis)
5. Testes BATS atualizados: simular falha em cada ferramentas e verificar que trap e acionado corretamente
6. Logs indicam FAIL no conteudo quando aplicavel (nao apenas no nome do arquivo)

## Dev Notes

### Pattern de Trap Handler (Orion Reference)

O SetupOrion NAO implementa trap handlers — isso e uma MELHORIA sobre o Orion, identificada na auditoria arquitetural da Aria (@architect). O pattern proposto segue boas praticas de bash scripting:

```bash
cleanup_on_fail() {
  local exit_code=$?
  if [[ $exit_code -ne 0 ]]; then
    echo -e "${UI_RED}Ferramenta falhou (exit code: $exit_code)${UI_NC}"
    echo -e "${UI_RED}Log: ${LOG_FILE:-'nao disponivel'}${UI_NC}"
    resumo_final 2>/dev/null || true
  fi
  log_finish 2>/dev/null || true
}
```

[Source: Auditoria @architect — Issues #1, #3, #21, #32]

### Compatibilidade com `set -euo pipefail`

**CUIDADO:** Varios scripts usam patterns que falham com `set -e`:
- `grep` retorna exit 1 quando nao encontra match — usar `grep ... || true`
- `docker ps --filter` pode retornar vazio — nao e erro
- `curl` pode falhar em network timeout — geralmente ja tem retry

O dev agent DEVE auditar cada script antes de adicionar `set -euo pipefail`.

[Source: deployer/lib/common.sh, deployer/ferramentas/*.sh — patterns existentes]

### Deployer.sh Subprocess Pattern

Cada ferramentas roda como subprocess: `bash "${SCRIPT_DIR}/ferramentas/XX-xxx.sh"`. Isso significa:
- O trap do ferramentas e isolado do deployer.sh
- O deployer.sh so ve o exit code final
- `$?` no deployer.sh reflete se o ferramentas terminou com sucesso

[Source: deployer/deployer.sh — linhas 53-102]
