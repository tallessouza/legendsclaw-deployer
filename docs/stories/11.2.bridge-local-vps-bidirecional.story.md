## Story

**As a** usuário que configurou minha máquina local via setup-local (story 11.1),
**I want** uma ferramenta que configure a bridge entre minha máquina local e o gateway OpenClaw na VPS,
**so that** eu consiga usar o Claude Code localmente com hooks que se comunicam com o gateway remoto via Tailscale, obtendo status real de saúde e funcionalidade completa da bridge.

## Description

Criar a ferramenta `setup-local-bridge.sh` que configura a conexão bridge entre a máquina local do usuário e o gateway OpenClaw rodando na VPS remota via Tailscale.

A ferramenta existente `12-bridge.sh` (Story 5.3) faz a configuração da bridge **na VPS** — conecta ao gateway local (`localhost:18789`). Esta story cria o equivalente **para a máquina local**, onde o gateway está remoto (acessível via Tailscale FQDN).

**Fluxo principal:**
1. Verifica que Tailscale está instalado e conectado
2. Coleta hostname Tailscale da VPS (interativo ou via auto-config)
3. Verifica conectividade via `tailscale ping`
4. Verifica saúde do gateway remoto via `curl` ao health endpoint
5. Cria service index com URL do gateway remoto (Tailscale FQDN)
6. Cria/atualiza `.claude/settings.json` com hooks — **merge sem sobrescrever**
7. Testa bridge.js localmente
8. Salva estado em `~/dados_vps/dados_bridge`

**Diferença chave vs 12-bridge.sh:** A URL do gateway usa `http://{hostname}.{tailnet}.ts.net:{porta}` em vez de `http://localhost:18789`.

[Source: deployer/ferramentas/12-bridge.sh]

### Bridge.js Core (Auto-Discovery)

O `bridge.js` em `.aios-core/infrastructure/services/bridge.js` funciona por auto-discovery — escaneia subdiretórios de `services/` procurando `index.js` com `name` e `health()`. **Não precisa ser modificado** — apenas o service index precisa apontar para o gateway remoto.

[Source: .aios-core/infrastructure/services/bridge.js]

### Settings.json Existente

O `.claude/settings.json` já contém hooks de bridge (SessionStart, PreToolUse, PostToolUse), permissions e configuração de idioma. A nova ferramenta **DEVE fazer merge** das hooks sem sobrescrever permissions e outras configs.

[Source: .claude/settings.json]

### Tailscale FQDN

O Tailscale usa FQDN no formato `{hostname}.{tailnet}.ts.net`. Pode ser obtido via:
```bash
tailscale status --json | jq -r '.Self.DNSName' | sed 's/\.$//'
# ou para obter o tailnet:
tailscale status --json | jq -r '.MagicDNSSuffix'
```

[Source: deployer/ferramentas/02-tailscale.sh]

## Scope

### IN
- Nova ferramenta `ferramentas/setup-local-bridge.sh`
- Verificação de Tailscale local → VPS (ping + health check)
- Coleta interativa ou auto-config de hostname/porta/agente
- Criação de service index com URL remota (Tailscale FQDN)
- Merge de hooks em `.claude/settings.json` (com backup)
- Health check remoto via Tailscale
- Testes bridge.js status/list
- State file e hints

### OUT
- Não instala Tailscale (isso é da story 11.1)
- Não configura o gateway no VPS (ferramentas 03/04 existentes fazem isso)
- Não altera bridge.js core (já funciona por auto-discovery)
- Não altera deployer.sh ou menu seletor (story 11.1)
- Não inicializa AIOS (story 11.3)

## Risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| Tailscale FQDN varia por tailnet (`.ts.net` suffix) | Medium | Detectar via `tailscale status --json` + fallback input manual |
| Firewalls corporativos podem bloquear Tailscale UDP | Medium | Hint sobre DERP relay (funciona via HTTPS) |
| settings.json merge pode corromper JSON | High | Backup .bak antes, usar `jq` para merge (ou Python se jq indisponível) |
| Gateway na VPS pode estar desligado durante setup | Low | Warning mas continuar — bridge reporta "offline" gracefully |
| Whitelabel pode não existir se começou pelo local | Medium | Coleta interativa como fallback |

### Bridge.js Auto-Discovery

```javascript
module.exports = {

  description: 'OpenClaw Gateway health for agent-name',
  health: async () => ({ status: 'ok'|'degraded'|'down', latency_ms, details })
}
```

[Source: .aios-core/infrastructure/services/bridge.js:20-51]

### Merge de settings.json (Ponto Crítico)

O `.claude/settings.json` atual tem ~87 linhas com `permissions`, `hooks` e `language`. O merge com `jq`:
```bash
jq '.hooks = $hooks' --argjson hooks '{"SessionStart":[...]}' settings.json > tmp && mv tmp settings.json
```
Se `jq` não disponível, fallback com Python:
```bash
python3 -c "import json,sys; d=json.load(open('$f')); d['hooks']=json.loads(sys.argv[1]); json.dump(d,open('$f','w'),indent=2)" '{"SessionStart":...}'
```
