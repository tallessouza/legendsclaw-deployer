# 9.2 — Extrair deployer para repo standalone legendsclaw-deployer (público)

## Story

**As a** operador que precisa instalar o Legendsclaw via `curl | bash`,
**I want** que o deployer viva num repositório público independente (`legendsclaw-deployer`),
**so that** o `install.sh` possa clonar direto de um raw URL público sem depender do monorepo privado.

## Description

Criar o repositório `legendsclaw-deployer` (público) no GitHub contendo apenas os arquivos necessários para o deploy funcionar de forma independente. O deployer já é praticamente standalone — as únicas referências externas (`.aios-core/`) são em 3 pontos e todas podem ser ajustadas ou tornadas opcionais.

O processo envolve:
1. Scan de secrets para garantir que nenhum `.env`, key ou password seja commitado
2. Cópia dos arquivos com estrutura flat (sem prefixo `deployer/`)
3. Ajuste das 3 referências externas a `.aios-core/`
4. Atualização do `REPO_URL` no `install.sh` para o novo repo
5. Criação do repo público + commit inicial + validação do raw URL

### Referências Externas Completas — Mapeamento Detalhado

**6 arquivos afetados, 3 patterns distintos:**

#### Pattern 1: `REPO_ROOT` + `.aios-core/` em `12-bridge.sh` (BLOCKER principal)
```bash
# Linha 12-13: navega para o parent do deployer
REPO_ROOT="$(cd "${SCRIPT_DIR}/../.." && pwd)"
cd "$REPO_ROOT"

# Linha 103: diretório dos services
SERVICES_DIR=".aios-core/infrastructure/services"

# Linha 168: path do bridge.js
BRIDGE_FILE="${SERVICES_DIR}/bridge.js"

# Linhas 199, 210, 221, 242, 253, 264: hooks do Claude Code referenciam bridge.js
"command": "node .aios-core/infrastructure/services/bridge.js status ..."
```
**Solução:** Remover `REPO_ROOT` (não faz sentido standalone). Mudar `SERVICES_DIR` para path relativo ao deployer: `"${SCRIPT_DIR}/../services"` ou `"${DEPLOYER_DIR}/services"`. O bridge.js pode ser colocado em `lib/bridge.js` ou `services/bridge.js`. Os hooks do Claude Code devem apontar para o novo path.
[Source: deployer/ferramentas/12-bridge.sh:12-13, 103, 168, 199-264]

#### Pattern 2: Fallback path em `15-validacao-final.sh`
```bash
# Linha 258: fallback se dados_bridge não tem Bridge File
bridge_file="${bridge_file:-.aios-core/infrastructure/services/bridge.js}"
```
**Solução:** Mudar fallback para `"lib/bridge.js"` ou `"services/bridge.js"` (novo path no standalone).
[Source: deployer/ferramentas/15-validacao-final.sh:258]

#### Pattern 3: Strings de hint em `lib/hints.sh` (cosmético)
```bash
# Linha 286: referência a agent file
echo "     Ou manualmente: .aios-core/development/agents/${nome_agente}.md"

# Linhas 707-708: comandos de debug do bridge
echo "    node .aios-core/infrastructure/services/bridge.js status"
echo "    node .aios-core/infrastructure/services/bridge.js list"

# Linha 721: require do service
echo "    node -e \"require('./.aios-core/infrastructure/services/${nome_agente}').health()...\""
```
**Solução:** Atualizar strings para novos paths. A referência à `.aios-core/development/agents/` pode ser removida (não relevante para deployer standalone).
[Source: deployer/lib/hints.sh:286, 707-708, 721]

### Paths Internos — Já Funcionam Flat

A maioria dos scripts usa path relativo que **já funciona** na estrutura flat:

```bash
# Pattern padrão (ferramentas/02-15):
SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
LIB_DIR="${SCRIPT_DIR}/../lib"
```

**Exceção — `01-base.sh`** usa pattern diferente:
```bash
SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")/.." && pwd)"
source "${SCRIPT_DIR}/lib/ui.sh"
```
Este também funciona flat pois navega de `ferramentas/` para o root.
[Source: deployer/ferramentas/01-base.sh:5]

### Fato Chave: Deployer Gera Tudo Inline

O deployer **não copia** arquivos do `.aios-core/`. Tudo é gerado via heredoc nos scripts:
- `05-whitelabel` → gera `apps/{agente}/config/`, `hooks/`, `lib/`, `skills/`
- `06-workspace` → gera SOUL.md, IDENTITY.md, USER.md, AGENTS.md, etc.
- `07-llm-router` → gera `llm-router-config.yaml`
- `08-skills` → gera `skills/{name}/index.js`
- `09-elicitation` → gera `elicitation/` skill completa

O `deployer/apps/mano/` é um **snapshot** de execução anterior — serve como referência/exemplo.
[Source: deployer/ferramentas/05-whitelabel.sh, 06-workspace.sh, 07-llm-router.sh, 08-skills.sh]

### Destinos na VPS (Runtime, Não Dependências)

Estes paths são **destinos** onde o deployer grava no target, não dependências:
- `~/.openclaw/workspace/` — 06 e 08 copiam workspace files pra lá (skip graceful se não existe)
- `/opt/openclaw/.env` — 08 popula com API keys (skip graceful)
- `$HOME/dados_vps/` — estado persistente do deployer
- `$HOME/legendsclaw-logs/` — logs de execução
- `$HOME/.clawd/memory/` — memory files exportados

Nenhum requer ajuste — são paths absolutos no target VPS.
[Source: deployer/lib/common.sh:11, deployer/lib/logger.sh:7, deployer/ferramentas/06-workspace.sh:558]

# Exceções esperadas (OK — são placeholders ou variáveis):
