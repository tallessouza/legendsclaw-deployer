## Story

**As a** operador que deseja automatizar o deploy completo do Legendsclaw,
**I want** um script runner (`deployer-auto.sh`) que execute todas as 15 ferramentas sequencialmente em AUTO_MODE, com timing, master log, relatorio final, e flags de controle,
**so that** eu possa executar deploys reproduziveis com um unico comando, retomar apos falhas, e ter visibilidade completa do resultado.

## Acceptance Criteria

1. `deployer/deployer-auto.sh` criado com flags: `--config PATH`, `--from NN`, `--to NN`, `--only NN,NN`, `--dry-run`
2. Valida existencia do config file antes de executar
3. Executa ferramentas na ordem sequencial (01→02→03→04→05→06→07→08→09→10→11→12→13→14→15)
4. Mede tempo de execucao (segundos) de cada ferramenta
5. Para na primeira falha com mensagem `--from NN` para retomar
6. `--dry-run` valida config e lista ferramentas sem executar
7. Master log em `~/legendsclaw-logs/auto-runner-{timestamp}.log`
8. Relatorio final com tabela OK/FAIL por ferramenta + totais + tempo total
9. `deployer.sh` — `run_ferramenta()` skip "Pressione ENTER" em AUTO_MODE
10. Exit code 0 se todos OK, 1 se algum FAIL

### Arquitetura do Runner

O `deployer-auto.sh` e um script **standalone** que:
1. Parseia flags de controle
2. Exporta `AUTO_MODE=true` e `AUTO_CONFIG=<path>`
3. Executa cada ferramenta como subprocess (`bash ferramentas/NN-*.sh`)
4. Cada ferramenta herda `AUTO_MODE` e `AUTO_CONFIG` do environment
5. Cada ferramenta faz seu proprio `auto_load_config` internamente
6. O runner captura exit codes e mede tempo

**CRITICO:** O runner NAO faz source das ferramentas — ele as executa como subprocessos independentes. Isso garante isolamento (cada ferramenta tem seu proprio `set -e`, trap handlers, etc).

### Patterns de Logging [Source: deployer/lib/logger.sh]

- Log dir: `~/legendsclaw-logs/`
- Log file pattern: `{nome}-{timestamp}.log` (ex: `auto-runner-20260221_143022.log`)
- Timestamp format: `%Y%m%d_%H%M%S`
- `exec > >(tee -a "$LOG_FILE") 2>&1` para capturar stdout+stderr
- Cada ferramenta cria seu proprio log (ex: `base-20260221_143030.log`) — o master log e adicional

### Patterns de UI [Source: deployer/lib/ui.sh]

O runner pode reutilizar as cores definidas em `ui.sh` ou definir suas proprias (o runner NAO faz source de ui.sh das ferramentas — e um script independente). Recomendado definir cores locais no runner para evitar dependencia.
