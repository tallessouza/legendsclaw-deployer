## Dev Notes

### Schema Supabase — Contexto Tecnico

Esta story cria a camada de persistencia necessaria para a skill elicitation (Story 4.2). As 3 tabelas sao consumidas pelo supabase-client.js da 4.2 via REST API com fetch nativo.

[Source: docs/prd/epic-4-skills-layer-elicitation.md#story-43]

### Supabase REST API Pattern

O acesso as tabelas e feito via REST API do Supabase com headers `apikey` e `Authorization` usando SUPABASE_SERVICE_ROLE_KEY. O service_role bypassa RLS por padrao, mas RLS deve ser habilitado por seguranca (defense in depth).

[Source: docs/architecture/legendsclaw-architecture.md#9-integrações-externas]

### Migration SQL Conventions

Seguir template padrao de migrations AIOS:
- Transacional: `BEGIN; ... COMMIT;`
- Idempotente: `CREATE TABLE IF NOT EXISTS`
- UUIDs como PK: `gen_random_uuid()`
- Audit columns: `created_at`, `updated_at` com trigger
- Indexes nomeados: `idx_{table}_{column}`
- Comments documentando tabelas e colunas
- Verificacao pos-migration com ASSERT

[Source: .aios-core/product/templates/tmpl-migration-script.sql]

### Seed Conventions

- Transacional: `BEGIN; ... COMMIT;`
- Idempotente: `ON CONFLICT (id) DO UPDATE SET ...`
- UUIDs fixos para seeds (reproducivel)
- JSONB cast explicito: `'...'::JSONB`
- Verificacao pos-seed com SELECT COUNT

[Source: .aios-core/product/templates/tmpl-seed-data.sql]

### Formato do Template JSONB (elicitation_templates.sections)

O campo `sections` da tabela `elicitation_templates` contem um array JSONB com o seguinte formato, compativel com o codigo da Story 4.2 (start_session, process_message, get_status):

```json
[
  {
    "name": "Founder & Story",
    "questions": [
      {
        "text": "Qual e o seu nome completo?",
        "type": "text",
        "required": true,
        "hints": "Nome do fundador principal"
      }
    ]
  }
]
```

Os campos `current_section` e `current_question` da tabela `elicitation_sessions` sao indices numericos (0-based) neste array.

[Source: docs/stories/4.2.skill-elicitation-estrutura-tools.story.md#tasks — Task 3.6, 4.6]

### Colunas Consumidas pela Story 4.2

O codigo da Story 4.2 referencia explicitamente:
- `elicitation_templates`: id, sections (JSONB)
- `elicitation_sessions`: id, template_id, status, current_section, current_question, responses (JSONB), created_at
- `elicitation_results`: id, session_id, template_id, data (JSONB), exported_at

As colunas adicionais (name, description, version, active em templates; updated_at em todas) sao para gestao e consulta — nao consumidas diretamente pelo codigo da 4.2.

[Source: docs/stories/4.2.skill-elicitation-estrutura-tools.story.md#tasks — Tasks 3, 4, 5, 6]

### Deployer Lifecycle Pattern

Seguir o lifecycle padrao de 8 etapas:
1. LOG INIT — log_init + step_init
2. LOAD STATE — dados() + verificar dependencias
3. INPUT — coletar SUPABASE_URL/KEY se necessario
4. CONFERINDO — confirmacao antes de aplicar
5. GERAR FILES — migration + seed via heredoc
6. HINTS — como aplicar no SQL Editor
7. VERIFICAR — curl REST para confirmar tabelas
8. FINALIZAR — dados_elicitation_schema + hints + log_finish

[Source: docs/architecture/legendsclaw-architecture.md#6-arquitetura-do-deployer-script]
