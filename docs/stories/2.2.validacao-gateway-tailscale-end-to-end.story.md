### Patterns do Deployer (Lifecycle)

Seguir o lifecycle simplificado (esta e uma ferramenta de validacao, nao de deploy):
1. LOG INIT — `log_init "validation-gw"` + step_init
2. LOAD STATE — `dados()` + verificar `dados_openclaw` e `dados_tailscale`
3. CHECK TAILSCALE — `tailscale status` (se falhar, FAIL mas continuar com `|| true`)
4. CHECK OPENCLAW — `systemctl is-active openclaw`
5. DISPLAY TEST COMMANDS — comandos para desktop
6. RUN DOCTOR — `openclaw doctor`
7. HEALTH CHECK — `curl localhost:{porta}/health`
8. MENSAGEM LOCAL — `pnpm openclaw agent --message "Teste"` (timeout 30s)
9. RECORD RESULTS — append em `dados_openclaw`
10. HINTS + RESUMO — hints contextuais por falha + resumo final

Nao ha resource gate (validacao nao consome recursos extras).
Nao ha input collection (todos os dados vem do estado salvo).
Script e executavel independentemente para re-validacao futura (nao requer menu entry).
Integrado como opcao pos-instalacao no 05-openclaw.sh.

[Source: docs/stories/2.1.ferramenta-openclaw-gateway-build-deploy.story.md#8-step-lifecycle]
