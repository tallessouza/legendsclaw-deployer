# 12.8 — Scripts de Validação Geral e Update

## Story

**As a** operador do deployer,
**I want** scripts de validação geral e update,
**so that** eu possa verificar o estado completo da instalação e atualizar configs quando necessário.

## Description

Criar dois scripts:
- `deployer/scripts/validate.sh` — validação completa: state files, configs, gateway, skills, workspace
- `deployer/scripts/update.sh` — backup + git pull + re-gera configs + re-valida

## Acceptance Criteria

- [ ] AC1: **validate.sh — State files:** Verifica que state files essenciais existem em `~/dados_vps/` (dados_whitelabel, dados_openclaw, dados_workspace)
- [ ] AC2: **validate.sh — Configs:** Chama `validate-config.sh` (Story 12.7) para validar JSON syntax, placeholders, .env permissions e secrets
- [ ] AC3: **validate.sh — Gateway:** Verifica gateway acessível via `curl -sf --max-time 5 http://localhost:{porta}/health`
- [ ] AC4: **validate.sh — Skills:** Verifica que diretório de skills existe e contém pelo menos uma skill com `SKILL.md`
- [ ] AC5: **validate.sh — Workspace:** Verifica que workspace do agente existe e contém os 8 bootstrap files obrigatórios
- [ ] AC6: **validate.sh — Exit code:** Retorna exit code 0 se todos os checks passam, 1 se algum FAIL
- [ ] AC7: **validate.sh — Resumo:** Exibe resumo final via `resumo_final` com contagem OK/SKIP/FAIL
- [ ] AC8: **update.sh — Backup:** Faz backup do diretório de configs antes de qualquer modificação (para `~/legendsclaw-backups/{timestamp}/`)
- [ ] AC9: **update.sh — Git pull:** Executa `git pull` no repositório do deployer; se falhar (conflitos), aborta com mensagem e não prossegue
- [ ] AC10: **update.sh — Re-validação:** Após pull, chama `validate-config.sh` para re-validar configs e `validate.sh` para validação geral
- [ ] AC11: Ambos os scripts usam libs do deployer para output consistente (ui.sh, logger.sh, common.sh)

## Scope

### IN
- `deployer/scripts/validate.sh`
- `deployer/scripts/update.sh`
- Output via ui.sh (step_ok/step_fail/step_skip)
- Testes BATS para ambos os scripts

### OUT
- Não executa ferramentas do deployer (apenas valida/atualiza)
- Não modifica state files
- Não re-gera configs (apenas re-valida via validate-config.sh)

## Risks

| Risco | Probabilidade | Impacto | Mitigação |
|-------|--------------|---------|-----------|
| `git pull` com conflitos em environment customizado | Média | Alto | update.sh aborta com mensagem clara; backup já feito antes do pull |
| Backup repetido consome espaço em disco | Baixa | Baixo | Naming com timestamp; operador pode limpar manualmente |
| Gateway offline durante validação (manutenção) | Média | Baixo | step_fail reporta mas não bloqueia outros checks |
| validate-config.sh (12.7) ainda não implementada | Alta | Alto | Dependência explícita; validate.sh verifica existência antes de chamar |

### Boilerplate dos Scripts

```bash
#!/usr/bin/env bash
set -euo pipefail

SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
DEPLOYER_DIR="${SCRIPT_DIR}/.."
LIB_DIR="${DEPLOYER_DIR}/lib"

source "${LIB_DIR}/ui.sh"
source "${LIB_DIR}/logger.sh"
source "${LIB_DIR}/common.sh"
```

### Libs Disponíveis — Funções Relevantes

**ui.sh:**
- `step_init N` — inicializa contadores com total de steps
- `step_ok "msg"` — imprime `[  OK  ]` e incrementa STEP_OK
- `step_fail "msg"` — imprime `[ FAIL ]` e incrementa STEP_FAIL
- `step_skip "msg"` — imprime `[ SKIP ]` e incrementa STEP_SKIP
- `resumo_final` — box com contagem OK/SKIP/FAIL
- Contadores globais: `STEP_CURRENT`, `STEP_TOTAL`, `STEP_OK`, `STEP_SKIP`, `STEP_FAIL`

**logger.sh:**
- `log_init "validate"` — cria `~/legendsclaw-logs/validate-{timestamp}.log`
- `log "msg"` — append timestamped line
- `log_finish` — fecha tee, imprime path do log
- Env var `LEGENDSCLAW_TEE_ACTIVE` — previne tee duplicado quando chamado como subscript

**common.sh:**
- `STATE_DIR="${REAL_HOME}/dados_vps"` — diretório dos state files
- `REAL_HOME` — resolve sudo para home do user real

### State Files — Referência

Todos em `~/dados_vps/`, formato `key: value`. Parse via:
```bash
valor=$(grep "Campo:" "$STATE_DIR/dados_xxx" 2>/dev/null | awk -F': ' '{print $2}' || true)
```

| State File | Campos Relevantes |
|------------|-------------------|
| `dados_whitelabel` | `Agente:`, `Display Name:`, `Apps Path:` |
| `dados_openclaw` | `URL Gateway:`, `Porta:`, `Install Path:` |
| `dados_workspace` | `Workspace Path:` |
| `dados_skills` | `Skills Ativas:` |
| `dados_gateway_config` | `Config Dir:`, `Status:` |

nome_agente="${1:-}"

### Gateway Health Check

```bash
gw_port=$(grep "Porta:" "$STATE_DIR/dados_openclaw" 2>/dev/null | awk -F': ' '{print $2}' || echo "19888")
for i in 1 2 3; do
  if curl -sf --max-time 5 "http://localhost:${gw_port}/health" >/dev/null 2>&1; then
    health_ok=true; break
  fi
  sleep 2
done
```

### validate-config.sh (Story 12.7) — O que faz

Script irmão que valida **configs específicos** do agente:
1. `.env` existe com permissão 600
2. Detecta `{{PLACEHOLDER}}` não resolvidos
3. Valida JSON syntax de `aiosbot.json`, `node.json`, `mcp-config.json`
4. Verifica variáveis obrigatórias no `.env`
5. Detecta secrets vazados em arquivos não-.env
6. Verifica portas disponíveis (18789, 55119)

**validate.sh (esta story)** é o wrapper que chama validate-config.sh + faz checks adicionais de runtime (gateway, skills, workspace, state files).

### Backup Pattern para update.sh

```bash
BACKUP_DIR="${REAL_HOME}/legendsclaw-backups/$(date +%Y-%m-%d-%H%M%S)"
mkdir -p "$BACKUP_DIR"
cp -r "${apps_path}/config" "$BACKUP_DIR/"
```

### Comportamento de Falha em update.sh

- **git pull falha:** step_fail + abortar imediatamente (configs não foram modificados, backup existe como safety net)
- **re-validação falha:** step_fail + continuar para mostrar resumo completo — NÃO fazer rollback automático (operador decide)

### Testing

**Framework:** BATS (Bash Automated Testing System)
**Localização:** `tests/deployer/`
**Runner:** `npx bats tests/deployer/{arquivo}.bats`

**Padrão de setup/teardown:**

```bash
#!/usr/bin/env bats

SCRIPT_DIR="$(cd "$(dirname "$BATS_TEST_FILENAME")/../../deployer" && pwd)"

setup() {
  source <(sed 's/^readonly //g' "$SCRIPT_DIR/lib/ui.sh" 2>/dev/null || true)
  source <(sed 's/^readonly //g' "$SCRIPT_DIR/lib/logger.sh" 2>/dev/null || true)
  source <(sed 's/^readonly //g' "$SCRIPT_DIR/lib/common.sh" 2>/dev/null || true)

  export STATE_DIR="$(mktemp -d)"
  mkdir -p "$STATE_DIR"
  export LOG_DIR="$(mktemp -d)"
}

teardown() {
  rm -rf "$STATE_DIR" "$LOG_DIR" 2>/dev/null || true
}
```

**Nota:** `sed 's/^readonly //g'` strip readonly dos constants para permitir override nos testes.

**Cenários de teste mínimos:**

**validate.bats:**
- Script existe e é executável
- Script tem `set -euo pipefail`
- Source libs corretas (ui.sh, logger.sh, common.sh)
- Com state files presentes → step_ok para cada
- Com state files ausentes → step_fail para cada
- Com workspace completo (8 files) → step_ok
- Com workspace incompleto → step_fail listando arquivos faltantes
- Exit code 0 quando tudo OK
- Exit code 1 quando algum FAIL

**update.bats:**
- Script existe e é executável
- Backup cria diretório com timestamp
- Backup copia configs corretamente
- git pull failure → aborta com step_fail
- Chamada a validate.sh após update
