# Validação Manual E2E — Story 11.2: Bridge Local→VPS

## Pré-requisitos da VM de teste

- [ ] Máquina Linux (Ubuntu 22+ ou similar)
- [ ] Tailscale instalado e conectado à mesma tailnet da VPS
- [ ] VPS com OpenClaw gateway rodando (porta 18789 ou configurada)
- [ ] Node.js >= 18
- [ ] jq instalado (ou python3 como fallback)
- [ ] Repositório legendsclaw clonado

## Checklist E2E (executar na VM de teste)

### Fase 1: Pré-execução

```bash
# 1. Confirmar Tailscale conectado
tailscale status
# Esperado: lista de peers, VPS visível

# 2. Confirmar gateway acessível
tailscale ping <hostname-vps> --timeout 10s -c 1
# Esperado: pong de <hostname>.tailnet.ts.net
```

- [ ] **PRE-1**: Tailscale status mostra a VPS como peer
- [ ] **PRE-2**: Tailscale ping retorna pong

### Fase 2: Execução do script

```bash
cd /caminho/legendsclaw
bash deployer/ferramentas/setup-local-bridge.sh
```

- [ ] **EXEC-1**: Script inicia sem erros (banner + step_init)
- [ ] **EXEC-2**: Detecta Node.js >= 18 (step_ok)
- [ ] **EXEC-3**: Detecta Tailscale instalado e conectado
- [ ] **EXEC-4**: Coleta hostname (interativo ou via auto-config)
- [ ] **EXEC-5**: Validação de hostname aceita input válido, rejeita inválido
- [ ] **EXEC-6**: Tailscale ping OK (step_ok) — AC3
- [ ] **EXEC-7**: Health check do gateway OK (step_ok com latência) — AC4
- [ ] **EXEC-8**: Service index criado em `.aios-core/infrastructure/services/{agent}/index.js` — AC5
- [ ] **EXEC-9**: Settings.json atualizado com hooks (backup .bak criado se já existia) — AC6, AC11
- [ ] **EXEC-10**: Bridge.js list mostra o serviço — AC8
- [ ] **EXEC-11**: Bridge.js status retorna status real do gateway — AC7
- [ ] **EXEC-12**: State file criado em `~/dados_vps/dados_bridge` — AC9
- [ ] **EXEC-13**: Hints exibidas (status + debug + next steps)
- [ ] **EXEC-14**: Resumo final exibido

### Fase 3: Verificação pós-execução

```bash
# 3. Service index aponta para Tailscale FQDN
cat .aios-core/infrastructure/services/*/index.js | grep "http://"
# Esperado: URL com .ts.net (não localhost)

# 4. Settings.json tem hooks
grep "bridge.js" .claude/settings.json
# Esperado: 3 referências (status, validate-call, log-execution)

# 5. State file com permissões corretas
ls -la ~/dados_vps/dados_bridge
# Esperado: -rw------- (600)
cat ~/dados_vps/dados_bridge
# Esperado: formato key: value com Gateway URL, Agente, etc.

# 6. Bridge status funcional
node .aios-core/infrastructure/services/bridge.js status
# Esperado: tabela com serviço(s) e status OK/DEGRADED

# 7. Bridge list funcional
node .aios-core/infrastructure/services/bridge.js list
# Esperado: lista com o agente configurado

# 8. Iniciar nova sessão Claude Code
claude
# Esperado: BRIDGE STATUS no início mostrando serviços com OK
```

- [ ] **POST-1**: URL no service index usa Tailscale FQDN (não localhost) — AC5
- [ ] **POST-2**: Settings.json contém 3 hooks de bridge — AC6
- [ ] **POST-3**: Settings.json preservou configs pré-existentes (permissions, idioma) — AC11
- [ ] **POST-4**: Backup .bak existe (se settings.json já existia antes) — AC11
- [ ] **POST-5**: State file com permissão 600 — AC9
- [ ] **POST-6**: State file contém todas as chaves esperadas — AC9
- [ ] **POST-7**: `node bridge.js status` retorna tabela com status real — AC7
- [ ] **POST-8**: `node bridge.js list` mostra serviço registrado — AC8
- [ ] **POST-9**: Nova sessão Claude Code exibe BRIDGE STATUS com OK

### Fase 4: Fallback offline — AC10

```bash
# 9. Desconectar Tailscale
sudo tailscale down

# 10. Re-executar o script
bash deployer/ferramentas/setup-local-bridge.sh
```

- [ ] **FALL-1**: Script NÃO crashou com Tailscale offline
- [ ] **FALL-2**: Step de conectividade mostra step_skip (não step_fail fatal)
- [ ] **FALL-3**: Restante do script executa (service index, hooks, state)
- [ ] **FALL-4**: Bridge.js status reporta "offline" sem erro fatal

```bash
# 11. Reconectar Tailscale
sudo tailscale up

# 12. Verificar que bridge volta
node .aios-core/infrastructure/services/bridge.js status
```

- [ ] **FALL-5**: Após reconectar, bridge.js status volta a funcionar

### Fase 5: Regressão VPS

- [ ] **REG-1**: As 16 ferramentas VPS (`01-base.sh` a `16-reload-agent.sh`) existem e não foram alteradas
- [ ] **REG-2**: `deployer.sh` não foi modificado
- [ ] **REG-3**: Testes existentes passam: `npx bats tests/deployer/`
