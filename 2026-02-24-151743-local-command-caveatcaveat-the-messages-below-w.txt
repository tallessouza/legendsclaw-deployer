  B. GAPS CRÍTICOS — O que o guide descreve e o deployer NÃO faz

  1. Criação do Agente no AIOS (Guide §4.4)

  O guide instrui criar .aios-core/development/agents/{agent-name}.md com
  definição YAML completa do agente. O deployer não toca no diretório
  .aios-core/development/agents/.

  O que falta:
  - Gerar o arquivo de definição do agente (persona, commands, dependencies)
  - Registrar o agente no sistema AIOS para que @{agent-name} funcione
  - Mapear tasks e workflows específicos do agente

  Impacto: Sem isto, o agente existe no OpenClaw mas não é reconhecido pelo 
  AIOS. Não pode ser ativado com @agent-name, não aparece no squad, não tem
  commands *.

  2. Cópia de Skills Existentes (Guide §6.1-6.3)

  O guide instrui copiar skills de apps/clawdbot/ como base. O deployer
  (ferramenta 08) cria skills do zero com templates mínimos, mas:

  O que falta:
  - Não copia hooks/session-digest/ (handler, ingester, scorer, templates,
  types)
  - Não copia lib/llm-router.js (a lib, não o config)
  - Não copia lib/metrics-alerts.js, metrics-collector.js, metrics-queries.js
  - Não copia skills avançadas como group-modes, memory (versão completa)
  - O index.js gerado pelo 08 é minimalista — não tem a estrutura modular do
  original

  Impacto: Skills ficam funcionais mas simplificadas. Faltam métricas, alertas
  inteligentes, session digest.

  3. Registrar Serviço no Bridge para Auto-Discovery (Guide §8.1)

  O deployer (ferramenta 12) já faz isto — cria o index.js em
  .aios-core/infrastructure/services/{agent}/. Porém há um gap sutil:

  O que funciona: Bridge detecta o serviço, faz health check.

  O que falta:
  - O bridge.js faz auto-discovery mas não registra o agente como consumidor de 
  skills — apenas monitora saúde
  - Não configura o roteamento de mensagens entre bridge e agente (quem recebe o
   quê)
  - A conexão é unidirecional (bridge → health check) sem canal de comando
  bidirecional

  4. Definição de Skill de Elicitation como Skill AIOS (Guide implica)

  O deployer (09) cria a skill de elicitation como módulo Node.js autônomo
  dentro de apps/{agent}/skills/elicitation/. Mas:

  O que falta:
  - Não registra a elicitation como skill AIOS com task definition em
  .aios-core/development/tasks/
  - Não cria o workflow AIOS que conecta elicitation → memory → agent context
  - Não define o elicit: true pattern que o AIOS framework usa para workflows
  interativos

  Impacto: A elicitation funciona no OpenClaw mas não é orquestrada pelo AIOS.
  Outros agentes não podem invocar *elicitate como comando.

  5. Configuração N8N + Portainer Stack (Guide §9)

  O deployer (01) configura Portainer e Traefik, mas:

  O que falta:
  - Não faz deploy da stack N8N completa (postgres, redis, workers, webhooks,
  editor)
  - Não configura os compose files para N8N
  - Não mapeia workflows N8N para skills do agente

  Impacto: N8N precisa ser instalado manualmente. O guide tem instruções
  detalhadas (§9.1-9.5) que o deployer ignora.

  6. Configuração de Canais Completa (Guide §10)

  O deployer (13) faz Evolution API + WhatsApp, mas:

  O que falta:
  - Não configura canal Slack (openclaw channel add slack)
  - Não configura webhooks ClickUp apontando para N8N
  - Não registra canais no OpenClaw via CLI (openclaw channel add)

  GAPS REAIS (confirmados no código)

  1. Setup Local do AIOS na máquina do usuário — NÃO EXISTE

  O deployer inteiro roda na VPS. Não há nenhuma ferramenta que:
  - Clone/instale o AIOS-Core no desktop/laptop do usuário
  - Configure o Claude Code local para conectar ao gateway remoto
  - Instale Tailscale no desktop
  - Configure hooks locais (.claude/settings.json no desktop)
  - Copie o bridge.js para a máquina local
  - Faça tailscale ping de validação local→VPS

  O install.sh existe, mas é um one-liner para VPS — clona o repo em
  /opt/legendsclaw e roda setup.sh. Não há equivalente local.

  Impacto: O usuário precisa fazer o setup local 100% manual. O guide menciona
  isso em §3.3 e §8.2 mas como passos manuais.

  2. Registro do Agente no AIOS — NÃO EXISTE

  Nenhum script cria .aios-core/development/agents/{agent-name}.md. O deployer
  cria:
  - apps/{agent}/ (estrutura OpenClaw) ✅
  - .aios-core/infrastructure/services/{agent}/ (bridge service) ✅
  - Workspace files em apps/{agent}/workspace/ ✅

  Mas não toca em .aios-core/development/agents/. Sem isto, o agente:
  - Não é ativável via @{agent-name} no AIOS
  - Não tem commands *help, *status, etc.
  - Não aparece na lista de agentes do squad

  3. Skills como Tasks AIOS — NÃO EXISTE

  As skills (08) e elicitation (09) são módulos Node.js autônomos dentro de
  apps/{agent}/skills/. Não há:
  - Task definitions em .aios-core/development/tasks/ para cada skill
  - Mapeamento skill OpenClaw → workflow AIOS
  - Integração da elicitation com o pattern elicit: true do framework

  4. Deploy N8N Stack — NÃO EXISTE

  O script 01 faz Traefik+Portainer, mas não há ferramenta para deploy completo
  do N8N (postgres, redis, workers, webhooks, editor). O guide descreve isso
  detalhadamente em §9.

  5. Canais além de WhatsApp — NÃO EXISTE

  O 13 faz Evolution/WhatsApp. Não há:
  - Slack channel setup (openclaw channel add slack)
  - ClickUp webhooks para N8N
  - Telegram setup

  6. hooks/session-digest/ são placeholders

  O script 05-whitelabel cria 7 arquivos em hooks/session-digest/ mas todos são 
  placeholders:
  // TODO: Implement session-digest handler
  module.exports = {};

  O mesmo para lib/llm-router.js, lib/metrics-alerts.js, etc. — todos
  module.exports = {};.

  7. lib/ são placeholders

  Os 4 arquivos em apps/{agent}/lib/ (llm-router.js, metrics-alerts.js,
  metrics-collector.js, metrics-queries.js) são todos stubs vazios. Não há
  ferramenta que os popule com código funcional.
